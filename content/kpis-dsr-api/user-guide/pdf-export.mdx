# 내보내기 및 제출

검증을 마친 공급내역 데이터를 KPIS에 제출하는 두 가지 방법(수동보고/자동보고)과 반송 처리 절차를 안내합니다.

## 수동보고: 엑셀 내보내기

KPIS 포털에 직접 업로드하려는 경우, 가공된 데이터를 엑셀 파일로 내보냅니다.

### 내보내기 절차

1. 데이터 편집 화면에서 **내보내기** 버튼을 클릭합니다.
2. 내보내기 범위를 선택합니다.
   - **현재 탭만**: 선택한 공급구분(출고/반품/폐기/정정/취소)의 데이터만 내보내기
   - **전체**: 모든 공급구분의 데이터를 한 파일에 내보내기
3. 갑지와 을지가 각각 별도 시트 또는 별도 파일로 생성됩니다.
4. 브라우저의 다운로드 폴더에 `.xlsx` 파일이 저장됩니다.

### 내보내기 파일 구성

| 항목 | 설명 |
|------|------|
| 파일 형식 | `.xlsx` (Excel 2007 이상) |
| 갑지 | 21열, 헤더 없음 — KPIS 업로드 양식과 동일 |
| 을지 | 3열, 헤더 없음 — 갑지 연번과 매칭 |

> 내보내기는 순수 읽기 작업입니다. DB에 저장된 데이터에 영향을 주지 않습니다.

### KPIS 포털 업로드

내보낸 엑셀 파일을 KPIS 포털(https://www.kpis.or.kr)에 직접 업로드하는 절차입니다.

1. KPIS 포털에 공동인증서로 로그인합니다.
2. **공급내역 보고** 메뉴에서 **엑셀 업로드**를 선택합니다.
3. 갑지 파일과 을지 파일을 각각 업로드합니다.
4. 제출 후 결과를 확인합니다.

## 자동보고: MA111 API 제출

시스템에서 KPIS API를 통해 직접 제출하는 방식입니다. 수동 업로드 없이 버튼 클릭으로 완료됩니다.

### 사전 조건

자동보고를 사용하려면 다음이 설정되어 있어야 합니다.

| 조건 | 설명 | 설정 위치 |
|------|------|----------|
| API 설정 | 사업자등록번호, API KEY 등록 | 관리자 > API 설정 |
| 공동인증서 | 유효한 `.pfx` 인증서 업로드 | 관리자 > 인증서 관리 |
| 유효성 검증 통과 | 오류 0건 | 데이터 편집 화면 |
| 사용자 승인 | 관리자가 승인한 계정 | 관리자 > 사용자 관리 |

> 공동인증서가 만료되었거나 미등록이면 전송 버튼이 비활성화됩니다.

### 제출 절차

1. 데이터 편집 화면에서 유효성 검증이 통과되었는지 확인합니다 (오류 0건).
2. **API 제출** 버튼을 클릭합니다.
3. 시스템이 MA111(공급내역 정보 직접 등록) API를 호출합니다.
   - 서버가 DB에서 공동인증서를 로드하고 복호화합니다.
   - SOAP XML을 생성하고 WS-Security 전자서명을 수행합니다.
   - KPIS 운영 서버에 데이터를 전송합니다.
4. 접수가 완료되면 각 행에 **접수번호**(`rcptNo`)가 부여됩니다.

### MA112 결과 확인

MA111로 제출한 데이터의 처리 결과를 확인합니다.

1. 제출 완료 후 **결과 확인** 버튼을 클릭합니다.
2. 시스템이 MA112(공급내역 등록 결과 조회) API를 호출합니다.
3. 각 건별로 처리 상태가 업데이트됩니다.

| 상태 | 의미 | 표시 |
|------|------|------|
| `submitted` | 접수 완료, 처리 대기 | 파란색 |
| `registered` | 등록 완료 | 초록색 |
| `returned` | 반송 — 수정 필요 | 빨간색 |

> KPIS 서버의 처리 시간에 따라 결과 조회까지 시간이 소요될 수 있습니다. 접수 직후에는 `submitted` 상태이며, 일정 시간 후 `registered` 또는 `returned`로 변경됩니다.

## 반송 처리

MA112 결과에서 반송(`returned`)된 건은 원인을 확인하고 수정한 후 다시 제출해야 합니다.

### 반송코드 (DC001~DC007)

| 코드 | 사유 | 원인 | 대응 방법 |
|------|------|------|----------|
| **DC001** | 표준코드 불일치 | 존재하지 않거나 만료된 표준코드 | MA101로 정확한 표준코드 재검증 후 수정 |
| **DC002** | 미등록 거래처 | KPIS에 등록되지 않은 거래처 | MA102로 거래처 등록여부 확인 후 사업자번호 수정 |
| **DC003** | 보고기간 외 공급일자 | 해당 분기 보고기간에 맞지 않는 날짜 | MA116으로 분기 보고기간 확인 후 공급일자 수정 |
| **DC004** | 중복보고 | 동일 데이터가 이미 등록됨 | MA112로 기등록 접수번호 확인 후 정정(4) 또는 취소(5) 처리 |
| **DC005** | 수량/금액 오류 | 단가 x 수량 ≠ 금액 | 단가, 수량, 금액 정합성 확인 후 수정 |
| **DC006** | 을지 누락 | 일련번호구분=0인데 을지 데이터 없음 | 을지 파일 재업로드 또는 일련번호구분 변경 |
| **DC007** | 요양기관기호 오류 | 공급형태=9(요양기관)인데 기관기호 누락/오류 | 8자리 요양기관기호 확인 후 수정 |

### 오류코드와 반송코드의 차이

| 구분 | 오류코드 | 반송코드 |
|------|---------|---------|
| 발생 시점 | API 호출 자체 실패 | 접수는 됐지만 검증 실패 |
| 원인 | 네트워크, 인증, 요청 형식 | 데이터 내용 문제 |
| 대응 | 인증서/네트워크 확인 | 데이터 수정 후 재제출 |

### MA113 재신청 절차

반송된 건을 수정한 후 재제출하는 절차입니다.

1. MA112 결과에서 반송(`returned`) 건을 확인합니다.
2. 반송코드(DC코드)를 확인하고 해당 데이터를 수정합니다.
3. **반송신청** 버튼을 클릭합니다.
4. 시스템이 MA113(반송신청 직접 등록) API를 호출합니다.
5. 접수 완료 후 MA114(반송요청 등록 결과 조회)로 결과를 확인합니다.

### 재신청 시 주의사항

- 반송신청은 반드시 **원래 접수번호**를 포함해야 합니다.
- 반송 사유에 맞는 필드만 수정합니다. 불필요한 필드 변경은 추가 반송의 원인이 됩니다.
- DC004(중복보고)의 경우 재신청이 아닌 정정(코드 4) 또는 취소(코드 5)로 처리해야 합니다.

## 제출 상태 요약

전체 데이터의 제출 현황은 사이드바의 달력에서 날짜별로 확인할 수 있습니다.

| 달력 상태 | 의미 |
|-----------|------|
| `done` (초록색) | 해당 날짜의 모든 건이 등록 완료 |
| `partial` (노란색) | 일부 미제출 또는 반송 건 존재 |
| `missing` (빨간색) | 데이터 없음 (미보고) |
| `ignored` (회색) | 미보고 무시 처리됨 |

# 문제 해결

StudioGo 운영 중 발생할 수 있는 문제와 해결 방법을 정리합니다.

## 일반적인 문제

### 로그인 실패

#### 증상

카카오 로그인 버튼을 탭해도 로그인이 완료되지 않거나 오류 메시지가 표시됩니다.

#### 원인 및 해결

| 원인 | 증상 | 해결 방법 |
|------|------|----------|
| 카카오 계정 미연동 | "카카오 계정을 확인해주세요" 메시지 | 카카오톡 앱에서 계정 로그인 상태 확인 |
| 카카오톡 미설치 (모바일) | 카카오 로그인 화면이 웹으로 열림 | 카카오톡 앱 설치 후 재시도 |
| 네트워크 오류 | "네트워크 오류" 또는 무한 로딩 | Wi-Fi/모바일 데이터 확인 후 재시도 |
| OAuth 토큰 만료 | 자동 로그인 실패 후 재로그인 요구 | 정상 동작. 카카오 계정으로 재로그인 |
| 앱 버전 미지원 | "앱을 업데이트해주세요" 메시지 | 앱스토어/플레이스토어에서 최신 버전 업데이트 |
| 서버 점검 중 | "서버 점검 중입니다" 메시지 | 공지 확인 후 점검 종료 시 재시도 |

#### 운영자 조치

사용자가 지속적으로 로그인에 실패한다고 보고하면 다음을 확인합니다.

1. Hono API 서버 상태 확인 (Vercel 대시보드)
2. 카카오 개발자 콘솔에서 OAuth 설정 확인 (Redirect URI, 앱 키)
3. Neon DB 연결 상태 확인

### 슬롯 충돌

#### 증상

BJ가 가용 상태의 슬롯을 선택했으나 "이미 예약된 슬롯입니다" 메시지가 표시됩니다.

#### 원인

두 명 이상의 사용자가 거의 동시에 같은 슬롯을 선택하는 경우 발생합니다. Hold 메커니즘이 동시성을 제어하지만, 극히 짧은 시간 차이로 충돌이 발생할 수 있습니다.

#### 해결 방법

| 대상 | 조치 |
|------|------|
| **BJ (사용자)** | 다른 가용 슬롯을 선택하여 재시도 |
| **운영자** | 별도 조치 불필요 (시스템이 자동 처리) |
| **관리자** | 빈번 발생 시 서버 로그에서 동시성 처리 확인 |

#### 예방

- 슬롯 선택 시 Hold가 즉시 걸리므로 정상적인 경우 충돌이 발생하지 않습니다
- 충돌이 반복된다면 서버 응답 지연이 원인일 수 있으므로 Vercel 함수 실행 시간을 모니터링합니다

### Hold 만료

#### 증상

슬롯을 선택하고 예약 정보를 입력하는 도중 "임시점유 시간이 만료되었습니다" 메시지가 표시됩니다.

#### 원인 및 해결

| 원인 | 해결 방법 |
|------|----------|
| 입력 시간 초과 (2분) | 슬롯을 다시 선택하여 재시도 |
| 네트워크 지연으로 서버 통신 실패 | 네트워크 확인 후 재시도 |
| 앱 백그라운드 전환 | Hold 중에는 앱을 전환하지 않도록 안내 |

#### 운영자 참고

Hold 만료율이 높다면 다음을 검토합니다.

- 예약 확정 화면의 입력 항목이 너무 많은지 확인 (간소화 검토)
- 서버 응답 시간이 느려서 실제 입력 가능 시간이 줄어드는지 확인
- 2분 제한이 적절한지 비즈니스 관점에서 재검토

### 푸시 알림 미수신

#### 증상

예약 상태 변경, 체크인 리마인더 등의 푸시 알림이 오지 않습니다.

#### 확인 사항

| 확인 항목 | 조치 |
|----------|------|
| 기기 알림 권한 | 설정 > StudioGo > 알림 허용 확인 |
| 앱 내 알림 설정 | StudioGo 앱 설정 > 알림 설정 확인 |
| 방해 금지 모드 | 기기의 방해 금지 모드 해제 |
| Expo 푸시 토큰 | 앱 재설치로 토큰 갱신 |

#### 관리자 조치

1. Expo 푸시 대시보드에서 발송 이력 확인
2. 실패한 푸시의 에러 코드 확인 (DeviceNotRegistered, InvalidCredentials 등)
3. 토큰 갱신 배치가 정상 동작하는지 확인

### 카카오 알림톡 미수신

#### 증상

알림톡이 카카오톡으로 수신되지 않습니다.

#### 확인 사항

| 확인 항목 | 조치 |
|----------|------|
| 카카오톡 설치 여부 | 카카오톡 미설치 시 알림톡 수신 불가 |
| 전화번호 일치 | 카카오 계정의 전화번호와 StudioGo 등록 번호 일치 확인 |
| 카카오 채널 차단 | 카카오톡에서 StudioGo 채널이 차단되지 않았는지 확인 |
| 알림톡 발송 한도 | 일일 발송 한도 초과 여부 확인 (관리자) |

### 예약 상태 불일치

#### 증상

앱에 표시되는 예약 상태와 실제 상태가 다릅니다. 예: 이미 승인된 예약이 앱에서는 여전히 대기로 표시됩니다.

#### 해결 방법

1. 앱을 완전히 종료하고 재시작합니다
2. 예약 목록 화면에서 아래로 당겨 새로고침합니다
3. 문제가 지속되면 앱 캐시를 삭제합니다

#### 관리자 조치

- API 응답 캐시 설정 확인 (캐시 TTL이 너무 긴 경우)
- DB 조회 쿼리에서 상태 필터링 오류 여부 확인

## 로그 확인 방법

### 서버 로그 (Vercel)

Hono API는 Vercel Serverless Functions로 배포되어 있습니다.

1. [Vercel 대시보드](https://vercel.com)에 로그인합니다
2. StudioGo 프로젝트를 선택합니다
3. **Logs** 탭에서 실시간 로그를 확인합니다

#### 주요 로그 필터

| 필터 | 용도 |
|------|------|
| `level:error` | 에러 로그만 필터링 |
| `path:/api/bookings` | 예약 관련 API 호출만 필터링 |
| `path:/api/auth` | 인증 관련 API 호출만 필터링 |
| `path:/api/slots` | 슬롯 관련 API 호출만 필터링 |
| `status:5xx` | 서버 에러 응답만 필터링 |

### DB 로그 (Neon)

1. [Neon 콘솔](https://console.neon.tech)에 로그인합니다
2. StudioGo 프로젝트를 선택합니다
3. **Monitoring** 탭에서 쿼리 성능과 연결 상태를 확인합니다

#### 주요 확인 항목

| 항목 | 정상 기준 | 이상 징후 |
|------|----------|----------|
| 연결 수 | 10개 미만 | 50개 이상이면 커넥션 풀 설정 확인 |
| 쿼리 응답 시간 | 100ms 미만 | 500ms 이상이면 인덱스 확인 |
| 에러율 | 0% | 1% 이상이면 쿼리/스키마 점검 |

### 앱 로그 (Expo)

개발 환경에서 Expo DevTools를 통해 클라이언트 로그를 확인할 수 있습니다.

```bash
# 개발 서버에서 로그 확인
npx expo start --dev-client
```

프로덕션 앱의 크래시 로그는 Expo Application Services(EAS)에서 확인합니다.

## 데이터 정합성 체크

### 정합성 문제 유형

| 유형 | 설명 | 영향 |
|------|------|------|
| 유령 Hold | Hold가 만료되었으나 DB에 남아 있음 | 가용 슬롯이 줄어 보임 |
| 상태 불일치 | 예약 상태와 슬롯 상태가 불일치 | 이중 예약 가능성 |
| 고아 예약 | 삭제된 스튜디오에 예약이 남아 있음 | 사용자 혼란 |
| 시간대 오류 | 서버/클라이언트 시간대 불일치 | 잘못된 시간에 슬롯 표시 |

### SQL 점검 쿼리

Neon 콘솔의 SQL Editor에서 다음 쿼리를 실행하여 정합성을 확인합니다.

#### 만료된 Hold 확인

```sql
-- 만료 시간이 지난 Hold 상태 슬롯 조회
SELECT s.id, s.studio_id, s.start_time, s.hold_expires_at
FROM slots s
WHERE s.status = 'HOLD'
  AND s.hold_expires_at < NOW();
```

#### 상태 불일치 확인

```sql
-- 예약은 취소인데 슬롯이 여전히 예약 상태인 건
SELECT b.id AS booking_id, s.id AS slot_id,
       b.status AS booking_status, s.status AS slot_status
FROM bookings b
JOIN slots s ON b.slot_id = s.id
WHERE b.status IN ('CANCELLED', 'REJECTED')
  AND s.status = 'BOOKED';
```

#### 고아 예약 확인

```sql
-- 삭제된 스튜디오에 남아 있는 미래 예약
SELECT b.id, b.studio_id, b.start_time, b.status
FROM bookings b
LEFT JOIN studios st ON b.studio_id = st.id
WHERE st.id IS NULL
  AND b.start_time > NOW();
```

### 정합성 복구

문제가 발견되면 다음 절차로 복구합니다.

1. **유령 Hold 정리**: 만료된 Hold를 AVAILABLE로 변경

```sql
UPDATE slots
SET status = 'AVAILABLE', hold_expires_at = NULL, hold_user_id = NULL
WHERE status = 'HOLD'
  AND hold_expires_at < NOW();
```

2. **상태 불일치 복구**: 취소/거절된 예약의 슬롯을 AVAILABLE로 변경

```sql
UPDATE slots s
SET status = 'AVAILABLE'
FROM bookings b
WHERE s.id = b.slot_id
  AND b.status IN ('CANCELLED', 'REJECTED')
  AND s.status = 'BOOKED';
```

3. **고아 예약 정리**: 삭제된 스튜디오의 미래 예약을 취소 처리

```sql
UPDATE bookings b
SET status = 'CANCELLED', cancelled_at = NOW(), cancel_reason = '스튜디오 삭제'
FROM (
  SELECT b2.id
  FROM bookings b2
  LEFT JOIN studios st ON b2.studio_id = st.id
  WHERE st.id IS NULL AND b2.start_time > NOW()
) orphan
WHERE b.id = orphan.id;
```

### 정기 점검 권장 주기

| 점검 항목 | 권장 주기 | 자동화 여부 |
|----------|----------|:----------:|
| 유령 Hold 정리 | 매 5분 | O (Cron Job) |
| 상태 불일치 확인 | 매일 새벽 3시 | O (Cron Job) |
| 고아 예약 확인 | 매주 월요일 | 수동 |
| DB 백업 확인 | 매일 | O (Neon 자동) |

## 긴급 연락처

### 장애 발생 시 연락 순서

| 순서 | 담당 | 역할 | 연락처 |
|:----:|------|------|--------|
| 1 | 시스템 관리자 | 서버/DB 장애 대응 | 내부 Slack `#studiogo-ops` |
| 2 | 백엔드 담당 | API/비즈니스 로직 이슈 | 내부 Slack `#studiogo-dev` |
| 3 | 카카오 기술지원 | 알림톡/OAuth 이슈 | [카카오 비즈니스 고객센터](https://business.kakao.com) |
| 4 | Vercel 지원 | 배포/서버리스 이슈 | [Vercel Support](https://vercel.com/support) |
| 5 | Neon 지원 | DB 연결/성능 이슈 | [Neon Support](https://neon.tech/docs/introduction/support) |
| 6 | Expo 지원 | 앱 빌드/푸시 이슈 | [Expo Support](https://expo.dev/support) |

### 장애 대응 절차

1. **증상 파악**: 어떤 기능이 영향 받는지 확인합니다
2. **범위 판단**: 전체 장애인지, 특정 기능 장애인지 확인합니다
3. **로그 확인**: Vercel Logs, Neon Monitoring에서 에러를 확인합니다
4. **1차 조치**: 가능한 경우 즉시 조치합니다 (재배포, 캐시 초기화 등)
5. **사용자 공지**: 장애 시간이 10분을 초과하면 인앱 공지를 등록합니다
6. **근본 원인 분석**: 장애 해소 후 원인을 분석하고 재발 방지 조치를 수립합니다
7. **포스트모템 작성**: 장애 타임라인, 원인, 조치 내역, 개선 사항을 기록합니다
